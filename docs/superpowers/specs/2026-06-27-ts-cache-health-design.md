# TS 全量缓存健康图设计

## 背景

当前后台已经能看到播放诊断、预取统计、最近分片样本和当前播放分片信息，但对于“当前播放头后方的连续缓存是否真的可用”仍然缺少直观证据。

现有监控更偏向摘要和最近样本，不能直接回答以下问题：

- 当前播放分片后面是否存在连续可用缓存
- 预下载是否形成了稳定的前向缓存带
- 缓存是否被过早淘汰，导致播放头前方出现空洞
- 某个具体分片是否正在下载、已经失败，或曾缓存但已失效

为此，后台需要新增一个按 ts 顺序展示的全量健康图，用于直接观察完整播放链路上的缓存状态分布。

## 目标

新增一个 `TS 全量健康图`，满足以下目标：

- 按分片顺序展示整个清单中的所有 ts
- 通过颜色直观展示每个 ts 的当前状态
- 明确标记当前播放分片
- 支持点击任意分片块后查看详细信息
- 替代一部分重复度高、信息密度低的现有摘要监控

本次设计不包含新的缓存策略，不调整预取算法，不引入持久化诊断数据。

## 非目标

以下内容不在本次范围内：

- 调整预取策略、并发策略、淘汰策略
- 新增顶部统计计数
- 为全量图增加刻度线或刻度数字
- 在块内显示每个分片的索引
- 追加持久化存储或历史回放能力
- 追加新的推荐操作、自动诊断建议

## 方案对比

### 方案 A：保留现有摘要视图，仅继续扩充数字指标

优点：

- 改动小
- 不需要新增完整分片状态表

缺点：

- 仍然无法直观看到播放头附近是否连续可用
- 仍然无法快速识别“缓存洞”
- 用户需要在多个数字和最近样本之间自己推断

### 方案 B：新增围绕播放头的窗口视图

优点：

- 聚焦当前最相关区域
- 渲染成本低

缺点：

- 看不到全局缓存推进情况
- 难以判断更远处分片是否已经被淘汰或失败
- 不符合当前排障目标

### 方案 C：新增 TS 全量健康图，并裁剪重复摘要

优点：

- 能直接看到整条播放链路上的缓存分布
- 最适合排查“下载很多但播放仍卡”的问题
- 可以明确看到当前播放头后方的连续性是否被打断

缺点：

- 需要新增每个分片的状态跟踪
- 前端和诊断数据结构都需要扩展

推荐采用方案 C。

## 界面设计

后台“播放排障”区域改为以 `TS 全量健康图` 为核心。

### 主图

主图按分片顺序平铺所有 ts，每个 ts 对应一个固定尺寸的块：

- 不显示块内索引
- 不显示顶部统计
- 不显示刻度线
- 当前播放块使用高优先级高亮
- 用户通过颜色观察播放头后续分片是否连续可用

颜色状态定义如下：

- 蓝色：当前播放分片
- 绿色：已缓存且仍有效
- 黄色：正在下载
- 灰色：曾缓存过但已被淘汰或失效
- 红色：下载失败
- 白色：尚未开始

状态优先级固定为：

1. 当前播放
2. 下载中
3. 下载失败
4. 已缓存
5. 已淘汰
6. 未开始

也就是说，当前播放分片即使已缓存，也显示为蓝色，不再同时显示绿色。

### 详情区

点击任意块后，在主图下方显示该分片详情：

- 分片索引
- 分片 URL
- 当前状态
- 是否已缓存
- 最近下载耗时
- 下载来源
- 最近失败原因
- 是否被淘汰

如果用户尚未选择任何分片，则详情区默认显示当前播放分片。

### 现有监控保留项

以下监控继续保留，作为主图下方的辅助信息：

- 当前播放分片索引
- 当前播放分片 URL
- 当前播放分片缓存
- 预取并发配置
- 当前 in-flight
- 当前等待预取数
- 已预取未消费数
- 预取领先量
- 缓存命中 / 未命中

### 现有监控移除项

以下监控从主界面移除：

- 最近请求分片
- 最近成功分片
- 最近失败分片
- 最近最慢 3 个分片
- 来源汇总
- 当前可见分片
- 原因汇总
- 当前主要问题

最近分片明细表可以保留为折叠区，供深挖时使用，但不再承担主视图职责。

## 数据模型设计

当前诊断状态只覆盖最近样本和少量聚合指标，不足以渲染全量健康图。

因此需要新增“按 manifest 维度维护的分片状态表”，最小跟踪字段包括：

- `index`
- `url`
- `state`
- `isCached`
- `wasEvicted`
- `isInFlight`
- `lastElapsedMs`
- `lastSource`
- `lastFailureReason`

建议状态表按分片索引组织，便于直接顺序渲染。

状态更新来源如下：

- manifest 解析阶段：建立完整分片列表，初始化为未开始
- 预取调度开始：标记对应分片为下载中
- 下载成功并写入缓存：标记为已缓存
- 下载失败：标记为失败并记录原因
- 缓存淘汰：如果该分片曾缓存成功，则标记为已淘汰
- 播放器实际请求分片：更新当前播放索引

为了支持灰色状态，缓存层在执行淘汰时需要回写对应分片状态，而不能只删除文件。

## 数据流

### 初始化

当加载 VOD manifest 时：

1. 解析所有 ts 条目
2. 创建该 manifest 的完整分片状态表
3. 所有分片初始状态为未开始
4. 将当前播放索引初始化为空或未开始

### 预取与请求阶段

当预取会话或实际播放触发分片下载时：

1. 分片进入下载中
2. 下载成功后更新为已缓存
3. 下载失败后更新为失败
4. 如果实际播放请求到该分片，更新当前播放索引

### 淘汰阶段

当缓存淘汰发生时：

1. 根据 manifestId 和 segmentIndex 定位分片
2. 如果该分片之前已缓存成功，则标记为已淘汰
3. `isCached` 改为 false
4. 保留 URL、最近耗时、失败原因等辅助信息

### 输出阶段

诊断快照中新增完整分片状态列表，后台页面直接渲染：

- 主图块列表
- 当前播放块高亮
- 详情区

## 错误处理

- 如果当前 manifest 未建立完整分片状态表，则主图显示“暂无全量分片状态”
- 如果某些分片缺少耗时、来源、失败原因，则详情区显示 `-`
- 如果缓存文件已不存在但未收到淘汰事件，前端不自行猜测为灰色，必须以后端状态为准
- 如果当前播放索引超出范围，前端不高亮任何块，并继续显示基础辅助信息

## 测试策略

测试分三层：

### 单元测试

- manifest 初始化后生成正确数量的分片状态
- 下载开始、成功、失败能正确推进状态
- 缓存淘汰后分片从“已缓存”转为“已淘汰”
- 当前播放分片高亮逻辑正确
- 状态优先级正确

### 页面测试

- 后台页面能渲染全量健康图
- 页面不再包含已移除的摘要项
- 点击块后详情区显示正确
- 当前播放块默认被选中或默认展示详情

### 集成测试

- manifest 加载后，诊断接口包含完整分片状态列表
- 实际播放请求会推动当前播放索引更新
- 预取成功和缓存淘汰会反映到诊断输出

## 实施顺序

建议按以下顺序实现：

1. 扩展后端诊断数据结构，支持完整分片状态表
2. 将预取、实际播放、缓存淘汰事件接入状态更新
3. 扩展诊断接口输出完整分片状态
4. 重构后台页面，加入全量健康图和详情区
5. 移除重复监控项，保留辅助信息
6. 更新测试并验证构建

## 风险与取舍

### 风险 1：分片数很大时页面密度过高

本次不显示索引、不显示刻度、不显示顶部统计，主图只承担状态展示，因此可接受较高密度。

### 风险 2：灰色状态不可靠

如果没有明确的淘汰事件回写，灰色状态会失真。因此缓存淘汰必须显式更新分片状态，而不能依赖前端推断。

### 风险 3：状态过多导致逻辑混乱

本次只保留 6 个状态，并规定显示优先级，避免后续继续堆叠视觉语义。

## 成功标准

当用户在电视上复现卡顿时，后台应能直接回答以下问题：

- 当前播放分片是否已缓存
- 当前播放头后方是否存在连续绿色缓存带
- 是否存在灰色淘汰块打断连续性
- 是否存在红色失败块或黄色长时间未完成块

如果这些问题能通过主图直接观察得到，则本次设计目标达成。
