# 管理后台分页与刷新边界设计

## 背景

当前管理后台虽然有左侧菜单，但本质上仍然是单页锚点式结构：

- 点击左侧菜单只是跳转到同一页面中的不同区块
- 日志刷新和诊断刷新脚本都注入在同一个页面中
- 缓存相关的运行态观察和代理/缓存配置混在一起

随着 `TS 全量健康图` 的引入，缓存相关内容已经变成后台最重的一块。如果继续维持单页堆叠，会带来三个问题：

- 页面结构越来越重，阅读和维护成本都高
- 实时刷新逻辑和静态配置逻辑混在同一页
- 用户在排障时需要在大量无关内容之间切换视线

因此需要把后台改成真正的分页结构，并明确哪些页面实时刷新，哪些页面只负责配置。

## 目标

本次设计目标：

- 将后台改造成真正分页结构，而不是单页锚点跳转
- 左侧菜单切换后，只显示当前页面内容
- 只在 `缓存` 和 `日志` 两个页面启用实时刷新
- 将代理配置、缓存配置、预取并发、诊断日志开关、清空缓存、APK 更新统一收拢到 `设置` 页
- 让 `缓存` 页专注运行态观察和卡顿排障

## 非目标

以下内容不在本次范围内：

- 新增用户权限、认证或多角色后台
- 改写底层缓存算法或代理算法
- 新增前端路由框架
- 持久化前端页面状态
- 新增实时推送机制（如 WebSocket）；本次仍基于现有轮询机制

## 已确认信息架构

左侧菜单固定为 4 页：

- `播放`
- `缓存`
- `日志`
- `设置`

其中：

- `缓存` 页实时刷新
- `日志` 页实时刷新
- `播放` 页不实时刷新
- `设置` 页不实时刷新

`更新` 页不再单独存在，合并进入 `设置` 页。

## 方案对比

### 方案 A：保留单页，仅隐藏/显示区块

优点：

- 改动小
- 可以复用现有锚点结构

缺点：

- 本质仍然是单页
- 页面初始化时仍要注入全部内容和大量无关脚本
- 不符合“真正分页切换，只显示当前页内容”的目标

### 方案 B：同一路由 + query 参数分页

例如 `/?page=cache`。

优点：

- 比单页切换更清晰
- 仍然只需要一个入口

缺点：

- 语义不如独立路径明确
- 测试和后续扩展时不如独立页面边界清晰

### 方案 C：独立路由分页

例如：

- `/play`
- `/cache`
- `/logs`
- `/settings`

优点：

- 真正分页
- 每页只渲染自身内容
- 每页可以按需注入脚本和刷新逻辑
- 路由语义清晰，便于测试和维护

缺点：

- 需要改造当前控制页构建入口

推荐采用方案 C。

## 页面职责设计

### 播放页

播放页只保留播放发起与停止相关内容：

- 输入播放地址
- 开始播放
- 停止播放

不做自动刷新。

### 缓存页

缓存页是主要运行态观察页面，负责：

- `TS 全量健康图`
- 分片详情
- 当前播放分片信息
- 预取/缓存辅助指标
- 上游竞速信息
- 最近分片明细

该页启用自动刷新。

缓存页不再承载代理配置、缓存配置和 APK 更新表单。

### 日志页

日志页只负责查看日志：

- 服务端日志
- 前端监控事件
- 复制日志
- 暂停 / 恢复日志刷新

该页启用自动刷新。

日志页不再承载缓存状态或配置项。

### 设置页

设置页统一收拢静态配置和管理操作：

- 代理配置
- 缓存配置
- 预取并发
- 诊断日志开关
- 清空缓存
- APK 更新

设置页不自动刷新。

用户在设置页执行表单提交时，仍然即时提交并显示结果反馈，但不会定时轮询更新页面内容。

## 页面内容布局

### 缓存页布局顺序

缓存页按以下顺序展示：

1. `TS 全量健康图`
2. `分片详情`
3. `当前播放分片信息`
4. `预取/缓存辅助指标`
5. `上游竞速`
6. `最近分片明细`

这样可以把最关键的运行态信号放在最上方，把辅助和深挖信息往下收。

### 设置页布局顺序

设置页按以下顺序展示：

1. 代理配置
2. 缓存配置
3. 预取并发
4. 诊断日志开关
5. 清空缓存
6. APK 更新

这样可以先放最常调的网络与缓存设置，再放低频的管理操作。

## 实时刷新边界

### 缓存页

缓存页自动轮询刷新：

- `diagnostics panel`
- `TS 全量健康图`
- 分片详情
- 当前播放分片信息
- 预取/缓存辅助指标
- 上游竞速

### 日志页

日志页自动轮询刷新：

- 服务端日志文本
- 前端监控事件合并视图

### 播放页与设置页

不做自动轮询。

说明：

- 这里的“不实时刷新”不是新增配置项
- 只是页面行为边界
- 表单提交后仍会立即返回结果并显示反馈

## 路由与渲染设计

建议把当前单一的 `buildControlPage(...)` 改成“共享外壳 + 按页面渲染主体”的结构。

### 共享外壳

共享外壳负责：

- 左侧菜单
- 页面标题
- 顶部状态信息
- 统一反馈区
- 统一样式

### 页面主体

按当前路由决定只渲染一个主体：

- `buildPlayPageContent(...)`
- `buildCachePageContent(...)`
- `buildLogsPageContent(...)`
- `buildSettingsPageContent(...)`

### 当前页高亮

左侧菜单需要高亮当前页，避免用户在分页后失去定位。

## 脚本注入策略

当前页面中已经混入多种刷新与交互脚本。分页之后应改成按页注入：

- `缓存` 页：只注入诊断刷新相关脚本
- `日志` 页：只注入日志刷新相关脚本
- `播放` 页：只注入表单提交反馈逻辑
- `设置` 页：只注入表单提交反馈逻辑

不要在 `播放` 或 `设置` 页保留日志/诊断刷新按钮、刷新状态文案或相关定时器。

## 测试策略

### 页面测试

需要验证：

- 左侧菜单链接不再是锚点，而是分页路由
- 每个页面只渲染对应主体内容
- `缓存` 页包含 `TS 全量健康图`
- `设置` 页包含代理、缓存、更新等配置表单
- `日志` 页包含日志自动刷新元素
- `播放` 和 `设置` 页不包含日志/诊断实时刷新元素

### 路由测试

需要验证：

- `/play`、`/cache`、`/logs`、`/settings` 返回 200
- 根路径可重定向或落到默认页面
- 当前页高亮正确

### 集成测试

需要验证：

- `缓存` 页仍能正确展示实时变化的诊断数据
- `日志` 页仍能正确展示实时日志
- 设置页表单操作仍然可用

## 风险与取舍

### 风险 1：共享状态散落到多个页面构建函数

应通过共享外壳和明确的数据输入结构控制复杂度，不要在每个页面里重复拼状态栏和反馈区。

### 风险 2：实时刷新脚本遗漏页面边界

分页后最容易出现的问题是：某页还残留别页的定时器或刷新按钮。实现时应显式按页注入，避免“全局都加载”的旧模式残留。

### 风险 3：缓存页再次变重

即便拆出设置页，缓存页仍然是重页面。因此缓存页必须保持“运行态优先”，不要再把配置表单重新塞回去。

## 成功标准

如果改造完成后满足以下条件，则本次设计达成：

- 点击左侧菜单后，只显示一个当前页面
- `缓存` 页专注运行态排障，且自动刷新
- `日志` 页专注日志查看，且自动刷新
- `设置` 页集中所有配置与管理操作，不自动刷新
- 页面之间职责清楚，不再出现单页堆叠和多套刷新逻辑混杂
